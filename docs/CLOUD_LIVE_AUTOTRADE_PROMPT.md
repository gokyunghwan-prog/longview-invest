# Longview 국내 소액계좌 클라우드 실전 자동투자 구현 프롬프트

## 목표

기존 국내 공시·시세 웹사이트를 손상하지 않으면서, PC가 꺼져 있어도 GitHub Actions가
매일 데이터를 갱신하고 한국투자증권 국내주식 전용계좌를 가치·장기 관점으로 자동
리밸런싱한다. 공개 웹의 **자동투자 선정순**과 실제 주문 엔진의 목표 종목·비중은 같은
검증된 산출물을 사용해야 한다.

## 절대 조건

1. 대상은 KOSPI·KOSDAQ 보통주와 KRW 현금뿐이다.
2. 목표 현금은 0%이나, 1주 단위·수수료·지정가·분산 상한 때문에 살 수 없는 잔돈은
   허용한다. 잔돈을 없애기 위해 종목·업종 집중도 상한을 깨지 않는다.
3. 약 100,000원 계좌에는 3~5종목, 종목·업종 최대 35%, 최소 포지션 20,000원,
   최소 주문 5,000원을 기본으로 한다. 100,000원을 넘는 고가 종목은 선정하지 않는다.
4. 웹과 주문 엔진은 `data/trading-selection.json`의 종목 ID와 목표비중을 함께 사용한다.
   이 파일에는 계좌번호, 실제 잔고, 수량, 주문번호, 체결내역을 넣지 않는다.
5. 기존 전체 연구 순위는 보존하되 웹 기본 정렬은 자동투자 선정순으로 한다.
6. KIS App Key, App Secret, 계좌번호, 상태 암호화 키는 GitHub Secrets로만 주입한다.
   `.env`, 로그, artifact, 공개 JSON, 커밋에 평문 비밀을 남기지 않는다.
7. GitHub Actions 임시 실행기의 로컬 파일이나 cache를 실전 원장으로 신뢰하지 않는다.

## 데이터와 선정 흐름

1. 매일 저녁 DART 공시와 금융위원회 국내 종가를 수집한다.
2. 기업 수·시세 커버리지·가격 기준일·모델 버전을 검증한다.
3. 동일한 가치·장기 점수 모델과 자동투자 안전 필터로 공개 선정 파일을 결정적으로 만든다.
4. 가격 검증 실패, 오래된 snapshot, 후보 부족이면 선정 상태를 `blocked`로 게시하고
   다음 날 주문을 전부 차단한다.
5. 선정 파일과 `companies.json`을 같은 검증 실행에서 커밋한다.

## 실전 주문 안전성

1. 다음 거래일 정규장 안에서만 최신 선정 파일을 사용한다.
2. KIS 현재가의 영업일자가 오늘과 다르면 휴장·오래된 시세로 보고 주문하지 않는다.
3. 주문 전에 KIS 미수 없는 매수가능금액·수량을 조회한다.
4. 실전계좌는 자동투자 전용이어야 하며 관리 밖 보유종목이 하나라도 있으면 차단한다.
5. 주문은 지정가만 허용하고 종목·업종·회전율·최대주문수·100,000원 자본 상한을 지킨다.
6. 주문 묶음 전에 원격 암호화 상태에 cycle과 전체 intent를 먼저 저장한다.
7. 주문 한 건마다 접수 결과와 KIS 주문번호를 즉시 원격 상태에 저장한 뒤 다음 주문으로
   진행한다. 저장 실패 시 후속 주문을 전송하지 않는다.
8. POST timeout이나 응답 불명확은 절대 자동 재시도하지 않고 `unknown`으로 차단한다.
9. 당일 주문·체결 조회로 submitted/partial/unknown을 조정한다. 미체결 취소 전에는
   정정취소가능수량을 조회한다.
10. 이전 실행이 불명확하면 새 매매를 전부 중단한다.

## 원격 상태

- GitHub 전용 `trade-state` 브랜치에 `state.enc` 하나만 저장한다.
- AES-256-GCM과 매번 새로운 nonce를 사용하고 32바이트 키는 GitHub Secret으로 둔다.
- GitHub Contents API의 기존 blob SHA를 사용한 CAS 저장만 허용한다.
- 409/422 충돌, 해시 불일치, 복호화 실패, 상태 크기 초과는 fail-closed 한다.
- 분산 lease와 fencing token을 상태에 포함한다. 주문 직전 원격 lease를 재확인한다.
- 공개 저장소에는 암호문 외 실제 보유·주문 상태를 커밋하지 않는다.

## GitHub Actions

- 데이터 갱신과 실전매매 workflow를 분리한다.
- 모든 외부 action은 검토된 commit SHA로 고정한다.
- 실전 workflow는 단일 concurrency group, `cancel-in-progress: false`를 쓴다.
- `pull_request`와 `pull_request_target`에서는 실전 Secrets와 주문 경로를 절대 사용하지 않는다.
- 예약 실행은 별도 활성 변수 없이는 주문하지 않는다.
- 수동 dry-run에서 인증·잔고·선정 parity·원격 상태 저장을 확인한 뒤에만 예약 실전을 켠다.
- Actions 예약은 지연·누락될 수 있으므로 오전 실행과 오후 조정 실행을 분리하고, 이전
  미결 상태가 있으면 다음 실행에서 먼저 대조한다.

## 필수 테스트

- 같은 snapshot과 정책에서 웹 선정 ID/비중과 엔진 목표 ID/비중이 완전히 같다.
- 100,000원에서 3~5종목을 선정하고 추가 매수가 불가능한 잔돈만 남긴다.
- 비싼 1위 종목은 명시적으로 제외되고 주문되지 않는다.
- KIS 매수가능금액보다 큰 주문과 미수 사용은 차단된다.
- 주문별 원격 checkpoint가 완료되기 전 다음 주문이 호출되지 않는다.
- 원격 CAS 충돌·lease 상실·runner 중단·KIS timeout에서 중복주문이 발생하지 않는다.
- 미체결·부분체결·취소·거절·unknown을 재조정하고 불명확하면 신규주문을 막는다.
- 비밀값이 오류, JSON, HTML, 로그, git diff에 나타나지 않는다.
- 기존 국내 순위·공시 API와 모의투자 테스트가 모두 통과한다.

## 완료 기준

실전 주문을 보내지 않는 cloud dry-run, 전체 자동 테스트, 공개 선정 parity, KIS 실전
인증·잔고·매수가능조회, 암호화 상태 CAS가 모두 성공해야 한다. 그 뒤에도 예약 실전은
GitHub 변수와 이중 동의문을 명시적으로 설정하기 전까지 비활성 상태여야 한다.
