# 자동 현금 투입 및 예약 실행 복구 디버깅 프롬프트

## 역할

당신은 실제 한국투자증권(KIS) 계좌를 다루는 장기·가치투자 자동화 시스템의
수석 디버거다. 기존 순위 사이트, 투자 판단 기준, 암호화 상태 저장, 주문
멱등성, GitHub Actions 자동화를 보존하면서 계좌의 투자 가능 현금이 장기간
남는 원인을 증거로 확인하고 수정한다.

이 작업은 새 투자 전략을 임의로 만드는 작업이 아니다. 웹사이트에 게시되는
국내 기업 순위와 자동투자 종목 선정을 계속 동일한 스냅샷과 기준으로 유지한다.

## 확인된 현상과 증거

1. KIS 읽기 전용 검증은 성공했다.
   - 실전 인증과 국내주식 잔고 조회가 모두 성공했다.
   - KIS 응답상 주문 가능 현금이 존재한다.
   - 따라서 핵심 원인은 API 키 누락이나 계좌 조회 실패가 아니다.
2. 2026-07-23 거래 예약 실행은 09:23 KST가 아니라 약 18:35 KST에 시작했다.
   - GitHub Actions run: `29995912427`
   - 계획 주문은 1건이었지만 실제 주문 전송은 0건이었다.
   - 이 시각은 시스템의 실전 주문 허용 시간인 09:05~14:50 KST 밖이다.
3. 2026-07-22 거래 예약 실행도 약 18:39 KST에 시작해 같은 방식으로
   주문 없이 차단됐다.
   - GitHub Actions run: `29908846099`
4. 최신 일일 데이터 수집은 예약 시각보다 약 5시간 30분 늦게 시작했다.
   - GitHub Actions run: `29936371518`
   - DART 연간 현금흐름 단계에서 `2630/2652`가 반복되며 6시간 후 취소됐다.
   - 그 결과 가격 검증, 순위 생성, 스냅샷 검증 및 커밋 단계가 실행되지 않았다.
5. 기존 일반 거래 계획은 최초 배치가 끝난 뒤 하루 최대 회전율 20%를
   적용한다.
   - 추가 입금은 새로운 최초 배치로 인식되지 않는다.
   - 현금 보유 목표가 0%여도 즉시 전액 투입을 뜻하지 않는다.
   - 정상 실행되더라도 큰 추가 입금은 이론상 최소 여러 거래일에 걸쳐 투입된다.
6. 별도 `topup` 명령은 수동 실행 전용이다.
   - 정기 워크플로는 이를 호출하지 않는다.
   - 이를 그대로 예약 실행하면 일반 종목 교체 규칙을 우회하고 최대 종목 수를
     초과할 수 있으므로 자동화 해법으로 사용하지 않는다.

## 최종 목표

1. 매 거래일 계좌에 새로 생긴 실제 투자 가능 현금을 자동으로 감지한다.
2. 기존 종목의 강제 매도나 전략 회전은 하루 20% 한도를 유지한다.
3. 시작 시점에 이미 존재하는 현금으로 목표 종목을 매수하는 부분은 별도
   매수 예산으로 처리해, 현금 목표가 0%인 전용 계좌에서는 가능한 범위까지
   같은 날 자동 배치한다.
4. 매수는 게시된 국내 순위와 동일한 검증 스냅샷을 사용한다.
5. 최대 종목 수, 종목당 비중, 최소 주문 금액, 섹터 제한, 교체 확인 횟수,
   관리 대상 종목 규칙을 모두 보존한다.
6. GitHub 예약 실행이 늦어지거나 한 번 누락돼도 장중 후속 실행이 같은
   일일 사이클을 안전하게 재시도한다.
7. 데이터 수집이 일부 DART 무응답 기업 때문에 매일 6시간씩 멈추지 않게 한다.
8. 공개 GitHub 로그에 계좌번호, API 키, 계좌 잔액, 보유 수량, 주문 상세를
   노출하지 않는다.

## 비목표

- 미국 기업 및 미국 주식 기능을 다시 추가하지 않는다.
- 투자 순위 알고리즘을 별개의 자동매매 알고리즘으로 분기하지 않는다.
- 매수 재원을 만들기 위해 당일 예상 매도 대금을 선사용하지 않는다.
- 예약 실행 문제를 숨기기 위해 주문 가능 시간을 확대하지 않는다.
- 개발 및 검증 과정에서 실제 주문을 전송하지 않는다.
- 수동 `topup`을 정기 실행으로 단순 전환하지 않는다.

## 구현 요구사항

### A. 일반 계획 안에서 추가 현금 자동 배치

1. `TRADING_AUTODEPLOY_CASH`와 같은 명시적 설정을 추가한다.
2. 실전에서 이 설정은 다음 조건을 모두 만족할 때만 허용한다.
   - 현금 목표 0%
   - 자동투자 전용 계좌 확인
   - 전용 계좌 자산 전체 사용 확인
   - 실전 주문 확인 문구
3. 계획기는 매 실행 시작 시점의 실제 투자 가능 현금을 기준으로 한다.
4. 매수·매도 회전 예산을 분리한다.
   - 매도 및 전략 교체: 기존 일일 최대 회전율 적용
   - 시작 현금으로 하는 목표 종목 매수: 가용 현금 범위까지 허용
5. 매수 한도 확대는 시작 현금에만 적용한다.
   - 예상 매도 대금은 KIS 당일 매수 재원으로 재사용하지 않는다.
6. 계획 결과가 목표 종목 수, 종목당 최대 비중, 최소 주문 금액과 충돌하면
   안전 규칙을 우선한다.
7. 배치 후 남은 현금이 있으면 공개 로그에는 금액 대신 아래와 같은
   비민감 원인 코드만 출력한다.
   - `BELOW_MIN_ORDER`
   - `CASH_LIMIT`
   - `POSITION_LIMIT`
   - `POSITION_WEIGHT_LIMIT`
   - `NO_ELIGIBLE_TARGET`
   - `UNRESOLVED_ORDER`
   - `OUTSIDE_ORDER_WINDOW`
   - `STALE_SIGNAL`

### B. 안전한 예약 재시도

1. 예약 전용 `auto` 명령을 추가한다.
2. `auto`는 GitHub API 응답의 신뢰 가능한 `Date` 헤더로 KST를 계산한다.
3. 09:05~14:50 KST 밖이면 KIS 계좌 조회나 상태 변경 없이 성공적인
   안전 건너뛰기로 끝낸다.
4. 장중이면 일반 `trade` 경로를 실행한다.
5. 같은 KST 거래일의 모든 예약 재시도는
   `scheduled-trade:YYYY-MM-DD` 같은 하나의 안정적인 사이클 키를 사용한다.
6. 첫 성공 실행이 끝난 뒤 같은 날 후속 예약은 주문 0건의 멱등 성공이어야 한다.
7. 첫 실행이 시간·데이터 문제로 주문 전에 차단됐다면 후속 실행은 다시
   시도할 수 있어야 한다.
8. 주문 의도가 저장된 뒤 중단됐으면 먼저 조정(reconcile)하고, 미확정 주문이
   남아 있으면 신규 주문을 차단한다.
9. 기존 원격 lease, fence, CAS, 주문 전 intent 저장, 주문별 checkpoint를
   그대로 유지한다.
10. GitHub 예약 실행은 정확한 시각을 보장하지 않으므로 평일에 여러 번
    heartbeat를 예약한다. 장외 heartbeat는 빠른 안전 건너뛰기여야 한다.

### C. DART 현금흐름 수집 정체 제거

1. 특정 공시에 현금흐름 데이터가 실제로 없는 결과도 검증된 음성 캐시로
   기록한다.
2. 동일 공시 ID이고 TTL이 지나지 않았다면 음성 캐시를 매일 다시 요청하지 않는다.
3. 공시 ID가 바뀌거나 TTL이 만료되면 다시 요청한다.
4. 과거의 손상된 캐시와 정상적인 `데이터 없음`을 구분할 수 있는 메타데이터를
   추가한다.
5. 현금흐름 보강 단계에 실행 시간 한도를 둔다.
   - 한도에 도달하면 현재 캐시를 안전하게 저장한다.
   - 다른 검증 가능한 재무 데이터로 순위 생성 파이프라인을 계속 진행한다.
6. 일일 수집 워크플로에는 서로 다른 시간대의 재시도 예약을 둔다.
7. 각 성공 수집은 웹사이트와 자동투자가 함께 사용할 당일 검증 revision을
   한 쌍으로 게시한다. 예약 재시도가 같은 원천 데이터를 다시 게시하더라도
   거래 사이클 멱등성으로 같은 날 주문을 중복 전송하지 않는다.

### D. 관측 가능성과 비밀 보호

1. 공개 로그에는 아래 정보만 허용한다.
   - 실행 명령
   - 성공/실패
   - 실제 주문 전송 여부
   - 계획 매수·매도 건수
   - 차단 원인 코드
   - 배치 단계
   - 멱등 중복 실행 여부
2. 공개 로그에서 금지할 정보:
   - API app key, secret, access token
   - 계좌번호 및 상품코드
   - 정확한 현금·총평가액
   - 보유 수량과 주문 가격·수량
3. 정확한 잔액 확인은 로컬 읽기 전용 검증에서만 표시한다.

## 반드시 유지할 안전 불변조건

1. 테스트와 개발 실행은 실제 주문 0건이어야 한다.
2. 실전 주문은 신뢰 시간 기준 09:05~14:50 KST에만 가능하다.
3. 순위 스냅샷과 자동투자 선택 파일의 revision 및 해시가 일치해야 한다.
4. 스냅샷이 오래됐거나 당일 검증 revision이 없으면 주문하지 않는다.
5. 미확정 주문이 있으면 신규 주문하지 않는다.
6. 관리 대상이 아닌 기존 보유 종목을 임의로 매도하지 않는다.
7. 한 번 완료된 일일 사이클은 재주문하지 않는다.
8. 공개 저장소 및 Actions 로그에 비밀이나 계좌 상세를 남기지 않는다.
9. 로컬 `.env`는 커밋하지 않는다.
10. 배포 전후 검증은 주문 전송이 불가능한 명령으로 수행한다.

## 필수 테스트 시나리오

1. 최초 배치 완료 상태에서 큰 추가 현금이 들어온 경우:
   - 기존 종목 매도는 20% 한도 안
   - 시작 현금 매수는 목표 비중과 안전 제한 안에서 20%를 넘어 가능
2. 시작 현금이 없고 매도 계획만 있는 경우:
   - 예상 매도 대금으로 같은 실행에서 매수하지 않음
3. 종목 수가 이미 최대인 경우:
   - 교체 확인 조건을 우회해 새 종목을 추가하지 않음
4. 최소 주문 금액보다 작은 현금:
   - 주문 없이 안전 종료하고 원인 코드 제공
5. 같은 날 첫 예약 성공 후 재시도:
   - 브로커 주문 호출 0건, 멱등 성공
6. 같은 날 첫 예약이 장외라 차단된 뒤 장중 재시도:
   - 정상 계획 가능
7. 장외 `auto`:
   - KIS 및 원격 상태 쓰기 호출 0건
8. 미확정 주문이 있는 재시도:
   - 조정 후 미해결이면 신규 주문 0건
9. DART `데이터 없음`:
   - 동일 공시·TTL 내 두 번째 실행에서 HTTP 재호출 없음
10. DART 공시 변경 또는 TTL 만료:
    - HTTP 재호출
11. 현금흐름 시간 한도:
    - 체크포인트 저장 후 전체 순위 파이프라인 계속
12. 전체 회귀 검사:
    - lint, 단위 테스트, 스냅샷/선택 일치 검사, 서버 smoke test 모두 통과

## 완료 기준

1. 상세 원인이 테스트로 재현되고 수정 후 테스트가 통과한다.
2. 전체 검사 명령이 성공한다.
3. 실전 주문 없이 KIS 읽기 전용 인증 및 계좌 조회가 성공한다.
4. 변경 사항은 별도 브랜치에 의도적으로 커밋된다.
5. GitHub에 push한 뒤 CI가 통과한다.
6. main 반영 후 예약 `auto`의 장외 실행이 안전 건너뛰기로 끝나는 것을 확인한다.
7. 다음 장중 실행부터 남은 투자 가능 현금을 자동 감지하고 동일 순위 기준으로
   배치할 수 있다.
8. GitHub Actions 자체의 지연 가능성은 제거할 수 없으므로, 다중 heartbeat와
   멱등 재시도로 영향을 줄였다는 제한을 사용자에게 명확히 알린다.
