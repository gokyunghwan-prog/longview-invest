# Longview 전체시장 구현 프롬프트 v2

> 이 문서는 전체시장 수집 기능을 구현한 당시의 기록입니다. 현재 점수 기준은
> `MASTER_PROMPT.md`의 가치·장기성장 모델 v2가 우선합니다.

아래 프롬프트는 기존 Longview를 소수 추적 종목용 화면에서 한국 KOSPI·KOSDAQ과
미국 주요 거래소 전체 상장기업용 공시 스크리너로 확장하기 위한 실행 명세다.

---

당신은 금융 데이터 엔지니어, 퀀트 리서처, 백엔드 엔지니어, 제품 디자이너다.
기존 Longview의 진실성 원칙과 공시 원문 추적성을 유지하면서 다음 요구사항을 모두
구현하라.

## 1. 목표와 포함 범위

- 한국은 Open DART에서 확인되는 현재 KOSPI(Y)·KOSDAQ(K) 상장법인을 모두 목록에
  포함한다. KONEX와 기타법인은 제외한다.
- 미국은 SEC의 `company_tickers_exchange.json`에서 Nasdaq, NYSE, CBOE로 표시된
  등록자를 포함한다. 같은 CIK의 복수 종목·주식종류는 한 회사로 묶고 전체 ticker를
  보존한다. OTC와 거래소가 없는 등록자는 제외한다.
- “전체”는 공식 원천에서 해당 거래소로 식별된 기업 전체를 뜻한다. ETF·SPAC·금융사,
  외국기업처럼 일반회사 모델을 신뢰성 있게 적용할 수 없는 대상도 검색 목록에서
  없애지 말고 `평가 보류` 또는 낮은 완전성으로 정직하게 표시한다.
- 공시 숫자가 부족한 회사를 임의 숫자로 채우지 않는다. 점수가 없거나 중립점에
  가까운 상태도 하나의 정상 결과다.

## 2. 한국 전체시장 수집

- 인증키는 서버 환경변수 `DART_API_KEY`에서만 읽고 로그·URL·결과 파일에 기록하지
  않는다.
- 매일 `corpCode.xml`을 확인해 현재 stock_code가 존재하는 법인을 기준으로 삼는다.
  이 파일에는 시장 구분이 없으므로 stock_code만으로 KOSPI·KOSDAQ을 추정하지 않는다.
- 최초 실행은 stock_code가 있는 각 법인의 `company.json`을 호출해 `corp_cls=Y/K`를
  확정하고 산업코드·결산월과 함께 체크포인트 캐시에 저장한다. 중단 후에는 이미 저장된
  회사를 다시 호출하지 않는다. 이후에는 corpCode의 modify_date가 변했거나 새로 생긴
  법인만 company.json을 재조회한다.
- `list.json`은 3개월 이하 구간, `corp_cls=Y/K`, `page_count=100`, 정정 포함 조건으로
  최근 공시를 증분 수집해 최신 공시와 정정·철회 표시를 갱신한다. corpCode에서 사라진
  stock_code는 활성 목록에서 제외하되 마지막 스냅샷을 즉시 삭제하지 말고 비활성
  이력을 남긴다.
- 재무는 기업별 호출이 아니라 고유번호를 최대 100개씩 묶어
  `fnlttMultiAcnt.json`과 `fnlttCmpnyIndx.json`을 사용한다. 재무지표 분류코드는
  M210000, M220000, M230000, M240000을 사용한다.
- 최근 제출 가능한 분기·반기·3분기 보고서를 우선하고 누락 기업은 직전 사업보고서
  11011로 폴백한다. 연결(CFS)을 우선하고 연결이 없을 때 별도(OFS)를 사용한다.
- 주요계정의 당기·전기·전전기 금액으로 매출, 영업이익, 순이익, 자산, 부채, 자본과
  3개년 추이를 만들고, 공식 재무지표에서 ROE·부채비율·유동비율·성장률을 교차 확인한다.
- 배치 API에 없는 영업현금흐름과 CAPEX는 값이 없는 것으로 처리한다. 상위 후보나
  변경 공시 기업만 `fnlttSinglAcntAll.json`으로 선택 보강할 수 있지만, 보강 실패가
  전체시장 목록 생성을 막아서는 안 된다.

## 3. 미국 전체시장 수집

- SEC 공식 대량 파일을 사용한다.
  - 회사·ticker·거래소: `https://www.sec.gov/files/company_tickers_exchange.json`
  - XBRL Company Facts: `https://www.sec.gov/Archives/edgar/daily-index/xbrl/companyfacts.zip`
  - 제출 이력: `https://www.sec.gov/Archives/edgar/daily-index/bulkdata/submissions.zip`
- 모든 요청에 `SEC_USER_AGENT`의 서비스명과 실제 연락 이메일을 전송한다. 재시도는
  지수 백오프를 사용하고, SEC 접근 정책을 우회하지 않는다.
- GitHub Actions가 매일 SEC 야간 재배포가 끝난 뒤 ZIP을 한 번씩 내려받는다. 수천 개
  회사별 API 호출로 대체하지 않는다.
- 압축 전체를 무조건 풀지 않는다. 공식 ticker 파일에서 선택된 CIK의 JSON만 읽고,
  점수에 필요한 US-GAAP 계정과 최근 연차 facts, 최근 주요 filing만 작은 중간 파일로
  가지치기한다.
- 10-K/10-K-A 외에 20-F/40-F 계열은 지원 가능한 taxonomy가 있을 때 계산한다.
  IFRS taxonomy 또는 필수 계정이 지원되지 않으면 회사는 목록에 유지하되
  `insufficient_data`로 표시한다.
- 동일 CIK의 여러 ticker는 한 기업 레코드로 만들고 대표 ticker와 `tickers[]`,
  `exchanges[]`를 함께 저장한다.
- 로컬 네트워크가 SEC에 403으로 차단되더라도 기존 미국 정상 스냅샷을 지우지 않는다.
  GitHub Actions에서도 같은 정책을 지키며 실패 보고서를 artifact로 남긴다.

## 4. 정규화와 점수

- 점수 모델 v1은 수익성 25, 성장성 20, 안정성 20, 현금흐름 20, 지속성 10,
  공시신뢰 5의 기존 100점 모델을 유지한다.
- 결측 배점을 다른 지표에 몰아주지 않는다.
  `최종점수 = 50 + (사용 가능 지표 원점수 - 50) × 완전성`으로 중립점에 수렴시킨다.
- 회사 카드에는 종합점수, 데이터 신뢰도, 완전성, 영역별 막대, 핵심 원지표,
  최근 재무 추이, 구조화 지표에서 고른 설명을 표시한다.
- 점수와 별도로 `live/stale/insufficient_data/not_applicable` 상태를 표시한다.
- 일반 모델이 부적합한 은행·보험·REIT·펀드 등은 전체 목록에는 포함하되 장기 검토
  후보에서는 제외하고 이유를 공개한다.
- PER·PBR은 현재 시세의 검증된 공급원과 기준시각이 없으므로 대시로 표시하고 점수에
  넣지 않는다.
- 모든 숫자는 provider, CIK 또는 corp_code, filing/접수번호, 보고기간, taxonomy,
  원문 URL로 추적 가능해야 한다.

## 5. 대규모 조회 API와 화면

- 브라우저에 전체 JSON을 보내거나 수천 개 카드를 한 번에 DOM에 만들지 않는다.
- `GET /api/overview`는 전체 회사 수, 국가별 수, 후보 수, 평균점수, 평균 신뢰도,
  산업 facet, 스냅샷 시각을 반환한다.
- `GET /api/companies`는 q, country, sector, sort, candidateOnly, page, pageSize를 받고
  기본 25개, 최대 100개만 반환한다. 동점 정렬은 id로 고정해 페이지가 흔들리지 않게
  한다.
- `GET /api/companies/:id`는 공시, 위험 플래그, 전체 lineage를 포함한 상세 레코드를
  반환한다.
- 서버는 스냅샷을 메모리에 한 번 로드해 검색 인덱스와 id Map을 만들고 파일 mtime이
  바뀌면 원자적으로 다시 로드한다. 파싱 실패 때는 마지막 정상 스냅샷을 제공한다.
- 검색 입력은 300ms debounce하고 이전 요청을 취소한다. 필터가 바뀌면 1페이지로
  돌아가며, 이전/다음과 페이지 번호를 키보드로 조작할 수 있어야 한다.
- 상세 정보는 카드를 눌렀을 때만 요청한다. 로딩·빈 결과·오류·stale·평가 보류를
  각각 디자인한다.

## 6. 일일 자동화와 저장

- GitHub Actions는 매일 오전 7시 15분 KST 이후 실행한다. SEC 야간 ZIP 완료 시각과
  시차를 고려해 미국 작업은 필요하면 더 늦은 별도 시각에 실행한다.
- 워크플로는 checkout, Node/Python 설정, 테스트, 한국 배치 동기화, SEC 벌크 다운로드,
  가지치기, 미국 정규화, 점수 재계산, 원자적 스냅샷 교체 순서로 실행한다.
- 지역 한쪽이 실패해도 다른 지역의 성공 결과와 실패 지역의 마지막 정상 결과를
  병합한다. 실패 종목 수천 개를 메인 JSON meta에 전부 넣지 말고 요약과 별도 진단
  파일로 나눈다.
- 자동 커밋은 데이터 파일만 대상으로 하며 코드·비밀값을 섞지 않는다. 대형 파일의
  Git 이력 증가를 감시하고, 저장소 한계를 넘기기 전에는 지역별 압축 snapshot 또는
  객체 저장소로 이전한다.
- GitHub Actions는 수집 작업자이지 동적 API 호스팅 서버가 아니다. 실제 공개 서비스는
  Node 서버가 실행되는 호스팅이 별도로 필요하다는 점을 README에 명시한다.
- 로컬 상시 서버 스케줄러와 GitHub Actions 중 하나만 주 수집기로 사용해 중복 호출을
  방지한다.

## 7. 실패 안전성과 완료 조건

- 동시 실행 잠금, 임시 파일 후 rename, 마지막 정상 스냅샷 보존, 공급자별 stale 표시를
  구현한다.
- 한 회사의 잘못된 XBRL이 전체 배치를 중단시키지 않아야 한다.
- 한국 전체시장 최초 구축과 일일 증분, 미국 ZIP 가지치기와 정규화는 각각 독립 실행할
  수 있어야 한다.
- 10,000개 합성 회사로 페이지·검색·정렬·상세 API를 테스트하고 목록 응답에 상세 공시
  배열이 섞이지 않는지 확인한다.
- 점수 결측, 안정적 동점 정렬, 잘못된 페이지 인자, 상세 404, 데이터 파일 교체 후
  reload, 공급자 실패 시 보존을 자동 테스트한다.
- README에는 사용자가 실제로 해야 할 일을 `npm start`, GitHub Secrets 등록,
  Actions 활성화로 구분해 짧고 정확하게 설명한다.
- 투자 권유가 아닌 정보·교육용 연구 도구라는 면책과 원문 확인 안내를 유지한다.

---

구현 우선순위는 전체시장 포함 여부, 데이터 계보, 결측의 정직한 표시, 일일 실패 복구,
페이지 성능, 화면 장식 순이다. 공식 공시로 확인할 수 없는 숫자를 만들어 완료율을
높이지 마라.
